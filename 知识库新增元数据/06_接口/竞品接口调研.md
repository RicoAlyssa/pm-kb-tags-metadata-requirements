# 竞品接口调研：知识库新增元数据

状态：进行中  
正式竞品：火山云/火山引擎、阿里云/百炼/PAI  
边界：Apifox/瀚海 OPEN-API 是我方底层接口源；金山云 API Explorer 是当前星流平台开放接口源；正式竞品只记录火山云/火山引擎、阿里云/百炼/PAI 的官方 API、官方文档与授权控制台/HAR 证据。

## 1. 结论先行

1. 元数据接口必须同时看“定义/赋值/检索过滤”三段。
   - 瀚海 OPEN-API 已看到元数据定义和文档元数据更新能力。
   - 金山云当前星流平台 API Explorer 尚未暴露同等元数据定义/赋值/过滤能力。
   - 竞品接口重点在“检索时怎么表达过滤条件”，这会反推 PRD 里的交互状态表。

2. 阿里云给出了两套有价值的接口形态。
   - 百炼 Model Studio：`Retrieve` 请求体里的 `searchFilters`，适合字段过滤和标签过滤。
   - PAI LangStudio：`RetrieveKnowledgeBase` 的 `MetaDataFilterConditions`，显式支持 `and/or` 与操作符。

3. 火山当前 UI 证据更强，API 证据待补。
   - 控制台可见强类型“文档标签”和内置 `doc_id` 过滤，已经能指导我方元数据入口和字段类型。
   - 官方 API 链接中有 `update_meta` 线索，但本轮网络环境打不开正文，因此不把字段级信息标为 VERIFIED。

## 3. 阿里云接口要点

### 3.1 Model Studio Retrieve + SearchFilters

官方示例中的最小结构：

```json
{
  "indexId": "o73yjlxxxx",
  "query": "公司中姓名为张三的员工",
  "searchFilters": [
    {
      "姓名": "张三"
    }
  ]
}
```

已确认语义：

- `SearchFilters` 用于在语义检索结果上继续过滤，尤其适合结构化数据。
- `searchFilters` 可包含多个子分组。
- 子分组之间默认 AND，且不可更改。
- 子分组内字段支持：
  - 单值查询：数值、字符串。
  - 多值查询：纯数值数组或纯字符串数组。
  - 范围查询：`eq`、`neq`、`gt`、`gte`、`lt`、`lte`。
  - 模糊查询：字符串 `like`。
  - 标签查询：仅文档搜索、音视频搜索类知识库。

对我方影响：

- 元数据过滤 PRD 必须写清楚运算符、值类型和组合关系。
- 如果我方后端只支持 AND，应在 UI 上显式提示；如果支持 OR，应比阿里更灵活。
- 元数据字段“是否参与检索”不只是展示字段，而是决定是否可进入过滤表达式。

### 3.2 PAI RetrieveKnowledgeBase + MetaDataFilterConditions

官方接口形态：

```http
POST /api/v1/langstudio/knowledgebases/{KnowledgeBaseId}/action/retrieve
```

核心请求字段：

| 字段 | 语义 |
|---|---|
| `WorkspaceId` | 知识库所在工作空间 ID |
| `Query` | 检索内容 |
| `TopK` | 返回排名最高的结果数 |
| `ScoreThreshold` | 相似度分数阈值，范围 `[0, 1]` |
| `MetaDataFilterConditions` | 元数据过滤条件，JSON 字符串 |
| `QueryMode` | `dense` 或 `hybrid` |

`MetaDataFilterConditions` 结构：

```json
{
  "FilterCondition": "and",
  "MetaDataFilters": [
    {
      "Key": "key1",
      "Value": "value1",
      "Operator": "=="
    },
    {
      "Key": "key2",
      "Value": "value2",
      "Operator": "!="
    },
    {
      "Key": "file_name",
      "Value": "prefix",
      "Operator": "contains"
    }
  ]
}
```

已确认语义：

- `FilterCondition` 支持 `and`、`or`。
- `Operator` 支持 `==`、`!=`、`contains`。
- `contains` 仅支持 `file_name` 字段。

对我方影响：

- 我方检索接口建议采用显式结构，而不是让前端拼 JSON 字符串。
- 如果为了快速兼容后端使用 JSON 字符串，也要在 PRD 写清楚前端如何校验和展示错误。
- 内置字段如文件名、文档 ID、来源 URL 应和客户自定义字段分层展示。

## 4. 火山接口/控制台要点

火山当前已验证的是 UI 与交互结构：

- 文档标签强类型字段非常接近元数据 Schema。
- `doc_id` 是无需配置即可用于检索过滤的内置字段。
- 检索/问答页面都存在按标签筛选能力。

官方 API 线索：

- `https://docs.volcengine.com/docs/84313/1269158?lang=zh`：项目已有证据索引记录其包含 `update_meta` 线索。
- `https://docs.volcengine.com/docs/84313/2192050?lang=zh`：项目已有证据索引记录其涉及标签权限过滤与 `doc_filter`。
- 本轮无法抓取火山正文，下一步需用授权浏览器或 HAR 复核字段。

下一轮必须抓的火山接口证据：

| 场景 | 需要抓取的请求 | 用途 | 脱敏要求 |
|---|---|---|---|
| 创建文档标签字段 | 创建知识库高级配置提交请求 | 确认字段类型枚举、字段名规则、选项结构 | 去除租户、账号、资源 ID |
| 编辑文档标签/元数据值 | 原始文档标签保存请求 | 确认文档级值更新结构 | 去除真实文档内容 |
| 检索筛选 | 知识检索/问答筛选请求 | 确认过滤表达式是否为 `doc_filter` 或其他字段 | 去除 Query、Cookie、Authorization |
| `update_meta` 官方文档 | 浏览器打开官方 API 页并截图/摘录 | 确认方法、路径、请求体、响应和错误码 | 不保存账号凭证 |
