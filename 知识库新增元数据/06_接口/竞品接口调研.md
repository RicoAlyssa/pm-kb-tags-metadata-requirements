状态：进行中（2026-07-14 已做官方后端文档复核，未发起真实请求）

## 1. 结论先行

1. 元数据接口必须同时看“定义/赋值/检索过滤”三段。
   - 瀚海 OPEN-API 已看到元数据定义和文档元数据更新能力。
   - 金山云当前星流平台 API Explorer 尚未暴露同等元数据定义/赋值/过滤能力。
   - 竞品接口重点在“检索时怎么表达过滤条件”，这会反推 PRD 里的交互状态表。

2. 阿里云给出了两套有价值的接口形态。
   - 百炼 Model Studio：`Retrieve` 请求体里的 `searchFilters`，适合字段过滤和标签过滤。
   - PAI LangStudio：`RetrieveKnowledgeBase` 的 `MetaDataFilterConditions`，显式支持 `and/or` 与操作符。

3. 火山已有控制台字段级检索请求体证据，文档值更新请求仍待补齐。
   - API 列表明确包含文档 `update_meta`，且说明文档 meta 信息更新会触发索引数据更新。
   - 独立 Playwright HAR 已确认检索过滤为 `Filter.Relation + Conditions[]`，每个条件含 `FieldName/FieldType/Operator/Values`，并设置 `FilterMode: "tag"`。
   - 本次未保存已有旗舰版文档的标签值，因此文档值更新请求体仍待测试资源上的保存采证。

## 3. 阿里云接口要点

### 3.1 Model Studio Retrieve + SearchFilters

官方示例中的最小结构：

```json
{
  "indexId": "o73yjlxxxx",
  "query": "公司中姓名为张三的员工",
  "searchFilters": [
    {
      "姓名": "张三"
    }
  ]
}
```

已确认语义：

- `SearchFilters` 用于在语义检索结果上继续过滤，尤其适合结构化数据。
- `searchFilters` 可包含多个子分组。
- 子分组之间默认 AND，且不可更改。
- 子分组内字段支持：
  - 单值查询：数值、字符串。
  - 多值查询：纯数值数组或纯字符串数组。
  - 范围查询：`eq`、`neq`、`gt`、`gte`、`lt`、`lte`。
  - 模糊查询：字符串 `like`。
  - 标签查询：仅文档搜索、音视频搜索类知识库。

对我方影响：

- 元数据过滤 PRD 必须写清楚运算符、值类型和组合关系。
- 如果我方后端只支持 AND，应在 UI 上显式提示；如果支持 OR，应比阿里更灵活。
- 元数据字段“是否参与检索”不只是展示字段，而是决定是否可进入过滤表达式。

### 3.2 PAI RetrieveKnowledgeBase + MetaDataFilterConditions

官方接口形态：

```http
POST /api/v1/langstudio/knowledgebases/{KnowledgeBaseId}/action/retrieve
```

核心请求字段：

| 字段 | 语义 |
|---|---|
| `WorkspaceId` | 知识库所在工作空间 ID |
| `Query` | 检索内容 |
| `TopK` | 返回排名最高的结果数 |
| `ScoreThreshold` | 相似度分数阈值，范围 `[0, 1]` |
| `MetaDataFilterConditions` | 元数据过滤条件，JSON 字符串 |
| `QueryMode` | `dense` 或 `hybrid` |

`MetaDataFilterConditions` 结构：

```json
{
  "FilterCondition": "and",
  "MetaDataFilters": [
    {
      "Key": "key1",
      "Value": "value1",
      "Operator": "=="
    },
    {
      "Key": "key2",
      "Value": "value2",
      "Operator": "!="
    },
    {
      "Key": "file_name",
      "Value": "prefix",
      "Operator": "contains"
    }
  ]
}
```

已确认语义：

- `FilterCondition` 支持 `and`、`or`。
- `Operator` 支持 `==`、`!=`、`contains`。
- `contains` 仅支持 `file_name` 字段。

对我方影响：

- 我方检索接口建议采用显式结构，而不是让前端拼 JSON 字符串。
- 如果为了快速兼容后端使用 JSON 字符串，也要在 PRD 写清楚前端如何校验和展示错误。
- 内置字段如文件名、文档 ID、来源 URL 应和客户自定义字段分层展示。

## 4. 火山接口/控制台要点

火山当前已验证的是 UI 与交互结构：

- 文档标签强类型字段非常接近元数据 Schema。
- `doc_id` 是无需配置即可用于检索过滤的内置字段。
- 检索/问答页面都存在按标签筛选能力。
- 标准版创建请求未携带标签 Schema，原始文档列表也没有标签列；旗舰版显示标签列和编辑入口。
- 检索过滤实际调用 `CreateSearchConfig`，随后调用 `GetKnowledgeServiceChat` 执行检索。

控制台实测过滤结构：

```json
{
  "Filter": {
    "Relation": 0,
    "Conditions": [
      {
        "FieldName": "liststring",
        "FieldType": "list<string>",
        "Operator": 0,
        "Values": ["product"]
      }
    ]
  },
  "FilterMode": "tag"
}
```

本次 UI 对应关系：`Relation: 0` = “且”；该 `List<string>` 条件的 `Operator: 0` = “包含”。其他枚举暂不外推。

官方 API/文档证据：

- `https://docs.volcengine.com/docs/82379/1511946?lang=zh`：API 列表包含文档 `update_meta`，且说明文档 meta 信息更新会自动触发索引数据更新。
- `https://docs.volcengine.com/docs/82379/1261891?lang=zh`：标签配置说明创建知识库时初始化标签、进入知识库详情原始文档页面后可编辑文档标签，标签配置后可用于检索过滤。
- `https://docs.volcengine.com/docs/82379/1353778?lang=zh`：知识检索参数说明支持按标签过滤文档范围，过滤条件包含/不包含/大于/小于/大于等于/小于等于，条件关系为且/或。

仍需补证：文档标签/元数据值保存请求、旗舰版创建标签 Schema 请求、错误码、权限码和批量语义。检索过滤字段级请求已完成采证。
