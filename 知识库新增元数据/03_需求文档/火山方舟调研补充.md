# PRD 补充：火山方舟元数据调研对需求的影响

更新时间：2026-07-10  
适用 PRD：知识库新增元数据

## 1. 必须进入 PRD 正文的结论

1. 火山未暴露独立“元数据”入口，但其“文档标签”已经具备元数据特征。
   - 支持字段名。
   - 支持字段类型。
   - 支持可选项。
   - 支持进入检索/问答过滤。

2. 我方仍应保持“元数据”独立命名。
   - 用户会议中描述的元数据是文档原始信息，例如 URL、客户自定义 key、关键业务信息。
   - 文档标签更偏分类/权限/业务分组。
   - 两者可以共用筛选组件，但对象和术语不要合并。

3. 内置字段需要单独建模。
   - 火山明确提示 `doc_id` 可直接用于检索过滤，无需配置。
   - 我方元数据 PRD 应区分“系统内置只读字段”和“客户自定义字段”。
   - 建议内置字段包括：doc_id、文件名、导入方式、上传时间、更新时间、目录路径、来源 URL（如系统可获得）。

4. 元数据过滤应进入检索/问答调试。
   - 条件行建议：字段名、运算符、输入值。
   - 支持添加多条件和 AND/OR。
   - 类型不同，运算符应不同：字符串支持包含/等于，数值支持比较，布尔支持等于，时间支持范围。

## 2. 字段类型建议

| 类型 | 来源 | 我方建议 |
|---|---|---|
| String | 火山字段类型下拉可见 | P0 |
| List<String> | 火山字段类型下拉可见 | P0，用于多枚举/多部门 |
| Int64 / Float32 | 火山字段类型与示例可见 | P0/P1，取决于我方接口是否已有数值类型 |
| Bool | 火山字段类型和示例可见 | P0 |
| Date_time | 火山字段类型下拉可见 | P0/P1，需确认格式和时区 |
| Geo_point | 火山字段类型下拉可见 | P2，除非客户场景明确需要 |

## 3. 建议改写的 PRD 模块

| PRD 模块 | 需要补充的内容 |
|---|---|
| 背景 | 加入“火山将强类型文档字段包装为标签；我方需独立定义元数据，避免和标签混淆” |
| 信息架构 | 增加“元数据定义管理”“文档详情元数据”“批量编辑元数据”“检索/问答元数据过滤” |
| 功能需求 | 增加字段类型、可选项、内置字段、只读字段、字段引用影响 |
| 交互 | 借鉴火山创建页的字段行：字段名 + 类型 + 选项；但放在元数据管理入口而不是只放创建页 |
| 验收 | 增加内置字段不用配置即可过滤；自定义字段定义后才可赋值/过滤 |
| 接口 | 与 Apifox/AIP 对齐：新增、更新、删除、查询元数据定义；更新文档 metadata；检索 metadata filter |

## 4. 与标签需求的边界

| 对象 | 标签需求 | 元数据需求 |
|---|---|---|
| 知识库级 | 业务/计费归属标签，涉及计费 | 不建议承担计费 |
| 文档级 | 分类、权限前置过滤、范围筛选 | 原始信息、自定义 key-value、检索过滤字段 |
| 类型体系 | 可以较轻，面向业务枚举 | 必须强类型，面向接口校验和过滤语义 |
| 展示位置 | 文档列表标签列、筛选范围 | 文档详情元数据区、列表可配置列、筛选范围 |
| API | tag filter / doc filter | metadata schema + metadata value + metadata filter |

## 5. 下一步接口核对重点

- AIP 中元数据定义是否是知识库级 schema，还是文档级自由 key。
- `id/name/value` 中 `id` 与 `name` 的关系：是否必填、是否可只传 name。
- value 的类型是否都是 string，还是支持强类型。
- 批量更新是覆盖、合并还是删除缺省字段。
- 检索过滤表达式是否支持 AND/OR、包含、等于、范围、空值。
