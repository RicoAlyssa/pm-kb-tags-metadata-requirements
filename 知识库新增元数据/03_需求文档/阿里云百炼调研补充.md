# 阿里云百炼调研补充：元数据需求影响

## 一句话结论

阿里云百炼把元数据做成了独立的“Meta 信息模板”，并在字段级拆出了“取值方法、参与检索、参与模型回复”三个关键控制点。我们做元数据时，不应该把它简化成标签，也不应该只做一个不可配置的 key-value 表。

## 对 PRD 的补充建议

| 模块 | 建议 |
| --- | --- |
| 入口 | 在知识库创建/编辑的索引配置中提供“元数据抽取/元数据模板”入口；已有知识库也要能查看模板和文档元数据值。 |
| 字段结构 | 每个字段至少包含字段 key、字段名称/描述、取值来源、默认值/配置、是否参与检索、是否参与模型回复。 |
| 取值方式 | P0 支持手动/常量、系统变量；P1 支持正则、关键词搜索；P1/P2 评估大模型抽取，需考虑成本、失败状态和人工修正。 |
| 检索能力 | “参与检索”字段应进入检索过滤条件白名单，避免所有元数据默认可筛选。 |
| 模型回复 | “参与模型回复”必须独立配置，默认建议关闭或按字段安全等级控制，避免 URL、客户信息、权限字段被模型直接输出。 |
| 文档列表 | 文档列表行内需要“元数据”入口，用于查看该文档当前元数据值和抽取状态。 |
| 历史文档 | 修改模板后是否重抽取历史文档必须在 PRD 写清楚：立即重抽、用户手动重抽、仅新文档生效三选一。 |

## 需要进入评审的问题

1. 元数据字段上限采用多少：阿里云截图可见 5 个字段，但我们的业务是否够用需要确认。
2. 元数据是否允许用户手工修正单个文档的值。
3. API 检索时元数据过滤语法由前端拼装还是后端提供查询 DSL。
4. 元数据字段是否需要权限等级，尤其是“参与模型回复”字段。
5. 大模型抽取是否进入首版范围；如果进入，需要明确计费、耗时和失败兜底。
