# 调研计划：知识库新增元数据

状态：执行中（公开官方文档证据 + 我方 Apifox 已知接口已纳入；控制台截图/Network 证据待采）

更新日期：2026-07-10

## 1. 调研目标

本需求的核心不是“有没有 metadata 这个词”，而是要确认一套能落地到控制台和检索链路的对象模型：

| 调研对象 | 要回答的问题 | 为什么影响当前需求 |
|---|---|---|
| 元数据定义 | 字段定义在知识库级还是文档级？支持哪些类型、名称规则、内置字段、自定义字段？ | 决定控台入口、接口字段、删除影响和数据一致性 |
| 文档元数据值 | 单文档/批量文档如何赋值、清空、覆盖、部分更新？ | 决定文档详情、文档列表批量操作和失败处理 |
| 检索过滤 | 元数据如何进入检索请求？支持等于、范围、包含、AND/OR 吗？ | 决定应用配置、召回测试和后端验收 |
| 删除/改名影响 | 删除字段、删除值、重命名字段对已有文档和检索有什么影响？ | 决定风险提示、审计和回滚 |
| 控台交互 | 入口放在知识库设置、文档详情、文档列表批量操作、召回测试哪里？ | 决定 PRD 交互表和 Pencil 原型 |

## 2. 当前已确认边界

- `VERIFIED`：元数据是文档级可扩展原始信息，可记录 URL 或客户自定义 key 信息。
- `VERIFIED`：检索时可携带元数据条件筛选文档/切片。
- `VERIFIED`：底层已有新增、更新、删除、查询元数据及更新文档元数据接口。
- `TO_CONFIRM`：元数据定义是否是知识库级 schema，或允许每篇文档临时新增 key。
- `TO_CONFIRM`：值类型、校验、唯一性、数量、长度、批量原子性、检索运算符。

## 5. 采证动作

### 5.1 控台页面

1. 找知识库设置中的元数据定义入口。
2. 创建 string/number/time 等字段，记录名称规则、类型锁定、重复名和删除影响。
3. 查看内置字段是否可开启/关闭，记录是否只读。
4. 在文档列表多选后批量编辑元数据，记录“应用到所有文档/仅已有字段”等策略。
5. 在文档详情编辑单文档元数据，记录保存、取消、清空、删除字段的差异。
6. 在召回测试或应用编排中配置 metadata filter，记录操作符和组合逻辑。

### 5.2 API / Network

1. 继续提取我方 Apifox 五个元数据接口的完整字段。
2. 若用户明确授权，再用测试环境发请求验证 200/400/401/403/404 和错误体。
3. Playwright 只保存脱敏 HAR，不保存 Cookie、Token、tenant/user 私有 ID。
