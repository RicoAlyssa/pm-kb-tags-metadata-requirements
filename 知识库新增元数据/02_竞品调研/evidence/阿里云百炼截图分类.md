## 二、已确认事实

1. 阿里云百炼把元数据能力明确命名为“Meta 信息 / Metadata 抽取 / Meta 信息模板”，不是把它隐藏在标签里。
2. Metadata 抽取的入口在创建知识库第 3 步“索引设置”里，开关开启后可进入模板配置。
3. 一个 Meta 信息模板最多可见 5 个字段，截图中出现 `添加字段 (1/5)`、`(2/5)`、`(3/5)`、`(5/5)`。
4. 字段配置结构为：
   - 字段（英文、下划线）
   - 中文描述
   - 取值方法
   - 值 / 配置入口
   - 是否参与检索
   - 是否参与模型回复
   - 操作
5. 取值方法可见 5 类：常量、变量、大模型、正则、关键词搜索。
6. 变量取值下拉中可见 `file_name` 和 `cat_name`。
7. 大模型取值不是直接填值，而是配置“实体描述”。
8. 正则取值需要输入正则表达式。
9. 关键词搜索支持多个关键词，分隔符为英文逗号。
10. 已有知识库中存在两个 Meta 入口：
    - 顶部按钮“Meta 信息”：查看/管理 Meta 信息模板。
    - 单文档行内“Meta 信息”：查看该文档对应字段和值。
11. “参与检索”和“参与模型回复”是两个独立开关，说明阿里云将“用于检索过滤/召回”和“进入模型回答上下文”拆成了两个控制点。

## 三、对我们元数据需求的启发

| 方向 | 可借鉴点 | 我们 PRD 中建议落法 |
| --- | --- | --- |
| 独立信息架构 | 元数据不是标签的附属能力，而是独立模板/字段体系。 | 在知识库设置或文档处理配置中设置“元数据模板”，避免和文档标签混在一个入口。 |
| 字段定义 | 英文字段名 + 中文描述，兼顾接口字段和控制台可读性。 | 字段 key 面向接口和检索，展示名/描述面向用户；字段 key 需校验英文、数字、下划线。 |
| 取值方式 | 常量、变量、大模型、正则、关键词搜索覆盖人工、系统、抽取三类来源。 | P0 至少支持手动/常量和系统变量；P1 再补正则、关键词、大模型抽取。 |
| 检索控制 | 是否参与检索独立开关。 | 每个元数据字段需要配置是否可作为检索过滤条件。 |
| 回复控制 | 是否参与模型回复独立开关。 | 每个字段需要配置是否暴露给模型上下文，避免敏感元数据默认进入回答。 |
| 文档级查看 | 文档列表行内可直接查看文档的 Meta 值。 | 文档列表应提供“元数据”入口，展示该文档当前字段和值，便于排查抽取结果。 |

## 四、待确认项

1. 字段名的完整校验规则：长度、是否允许数字开头、是否允许重复。
2. 模板字段数上限是否固定为 5，还是不同版本/套餐可配置。
3. “参与检索”的真实语义：是仅用于检索召回，还是也可作为外部 API filter 条件。
4. “参与模型回复”的真实语义：是否把字段值拼进上下文、引用来源、答案补充信息，还是只影响模型内部可见性。
5. 变量取值是否只有 `file_name`、`cat_name`，是否还有 URL、上传人、文档 ID、时间等系统变量。
6. 大模型抽取的成本、异步状态、失败重试、人工修正能力。
7. 已有知识库修改模板后，是否会触发历史文档重解析/重抽取。
8. Meta 信息模板是否支持 API 创建、更新、删除，以及字段是否在检索 API 返回。
