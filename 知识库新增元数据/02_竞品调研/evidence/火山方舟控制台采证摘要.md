## 2. 对我方“元数据”需求的建议

1. 元数据不要被标签需求吞掉。
   - 火山的“文档标签”已经承担了强类型字段能力，但用户原始需求中“元数据”强调原始信息、来源 URL、自定义 key。
   - 我方可以复用火山的字段定义交互，但 PRD 中仍应把对象命名为“元数据字段/metadata schema”。

2. 建议拆成两类字段。
   - 内置只读字段：`doc_id`、文件名、上传时间、导入方式、来源 URL（如有）、目录路径等。
   - 客户自定义字段：客户自己定义 key、类型、可选项，并给文档赋值。

3. 字段类型建议先按 P0/P1 分层。
   - P0：String、List<String>、Number/Float、Boolean、DateTime。
   - P1：List<Number>、GeoPoint、枚举模板、字段引用统计。
   - 具体类型命名、空值语义、数组语义必须以我方 AIP/后端接口为准。

4. 检索/问答过滤组件要统一。
   - 条件行：字段名/标签名、运算符、输入值。
   - 组合关系：AND/OR。
   - 内置字段 `doc_id` 可默认可用；自定义字段只有定义后才可选。

## 3. 与火山差异化的产品判断

| 主题 | 火山做法 | 我方建议 |
|---|---|---|
| 命名 | 统一叫“文档标签”，但字段类型很像元数据 | UI 上清晰区分“标签”和“元数据”，避免 PM、研发、客户理解混乱 |
| 字段定义时机 | 创建知识库时定义文档标签；创建后是否可改需继续验证 | 我方应支持在知识库详情设置中持续维护元数据字段，且有影响范围提示 |
| 内置字段 | `doc_id` 无需配置即可过滤 | 我方应显式展示内置字段清单与只读规则 |
| 检索入口 | 检索页用 @，问答页用右侧筛选范围 | 我方可以统一为“检索范围/过滤条件”组件，减少学习成本 |
