# 竞品横向对比：知识库新增标签

状态：初版（公开官方文档证据已覆盖 Dify；控制台截图待采）

## 1. 快速结论

- Dify 已有“知识库级标签”的公开 API：标签 CRUD、绑定/解绑知识库、按标签查看知识库标签；这对我方知识库级标签的对象建模有参考价值。
- Dify 公开文档中的标签类型始终是 `knowledge`，目前公开证据只覆盖知识库级标签，不覆盖我方“文档级标签”。
- Dify、阿里云等竞品会用 metadata 做文档过滤；这能借鉴“检索前过滤链路”，但不能把我方文档级标签直接合并到元数据需求里。
- 目前没有公开证据证明 Dify 知识库标签与计费/成本中心绑定；我方知识库级标签的计费链路需要按业务平台重新设计。

## 2. 横向对比表

| 维度 | 我方目标 | Dify 公开证据 | 差异/启发 | 结论状态 |
|---|---|---|---|---|
| 标签对象层级 | 分为知识库级标签、文档级标签 | Dify Tags API 文档显示标签类型为知识库标签，绑定目标是 knowledge base/dataset | 可参考知识库级标签；文档级标签仍需单独找竞品和我方接口证据 | VERIFIED / TO_CONFIRM |
| 知识库级标签 CRUD | 创建、编辑、删除、查询，需有使用量和删除影响 | Dify 提供创建、列表、更新、删除知识库标签 API；公开字段含 `id/name/type/binding_count` | 我方可保留使用量/绑定数作为删除提示和列表辅助字段，但字段名以我方接口为准 | VERIFIED |
| 知识库绑定/解绑 | 一个知识库可绑定多个标签；支持批量绑定/解绑 | Dify Create Tag Binding 支持 `tag_ids[]` 和 `target_id`，解绑也支持 `tag_ids[]` | 我方可借鉴批量绑定接口形态，但需补计费同步、审计和失败补偿 | VERIFIED |
| 知识库列表筛选 | 按标签筛选知识库，用于应用/AI 路由 | Dify List Knowledge Bases 文档说明支持 tags 过滤，响应示例含 `tags` | 我方需要进一步定义：筛选发生在控制台列表、应用编排，还是检索路由服务 | VERIFIED / TO_CONFIRM |
| 计费/业务归属 | 知识库级标签必须接入计费/项目/业务归属 | Dify 公开标签 API 未体现计费字段；公开证据不足以证明其支持 | 我方不能照搬 Dify 简化模型，必须新增业务平台对接设计 | TO_CONFIRM |
| 文档级标签 | 文档打标，用于检索前过滤文档/切片，不涉及计费 | 当前公开 Dify 标签证据未覆盖文档级标签；Dify 文档过滤主要以 metadata 体现 | 我方文档级标签需要单独定义对象、入口、批量操作和检索条件 | TO_CONFIRM |
| 权限关系 | 标签可能作为过滤条件，但不等同完整权限系统 | 公开证据暂未证明标签直接承担权限 | PRD 中需把“标签过滤”和“权限授权”分层，避免标签过载 | TO_CONFIRM |
| 删除/改名生效 | 删除或改名需提示绑定对象、检索生效和计费影响 | Dify 公开字段有 `binding_count`，但删除影响细节需控制台确认 | 我方应要求删除前展示引用对象，知识库级额外展示计费/应用影响 | INFERRED |
| API 安全 | 不在前端保存高权限密钥 | Dify API 文档提示 Knowledge API key 可访问创建者可见知识库，需谨慎处理 | 我方若开放标签 API，也应区分控制台权限和服务 API 权限 | VERIFIED |

## 4. 对 PRD 的建议

1. PRD 保持“两类标签两章”：知识库级标签负责路由 + 计费归属；文档级标签负责检索过滤。
2. 知识库级标签的 P0 必须包含业务平台同步、计费归属、失败补偿和审计。
3. 文档级标签的 P0 不出现计费字段，只覆盖打标、批量、过滤、生效、权限边界。
4. 可以借鉴 Dify 的 `binding_count` 思路，但我方字段必须等底层/业务平台接口确认。
